# kafko v0.1.1 criterion baseline — 2026-05-25

First criterion measurement run after the v0.1.1 work landed (uncommitted at
time of capture). Records the headline post-`send_batch` numbers and the
`LogConfig` sweep results. **Criterion data also pinned under
`target/criterion/<group>/<param>/v0_1_1/`** via
`cargo bench -- --save-baseline v0_1_1` so future runs can A/B against it
with `cargo bench -- --baseline v0_1_1`.

## Headline #1 — `send_batch` vs loop of `send()` (256 B values, no compression)

`partition.append_batch(N)` vs a loop of N `partition.append()` calls at the
same per-record size. The gap is the mpsc round-trip cost saved per record.

| N | `send_batch_none` | `send_loop_none` | Speedup |
|---:|---:|---:|---:|
| 1 | 273 K rec/s | 272 K rec/s | 1.0× |
| 8 | 635 K rec/s | 261 K rec/s | **2.4×** |
| 32 | 1.34 M rec/s | 241 K rec/s | **5.6×** |
| 128 | 2.26 M rec/s | 245 K rec/s | **9.2×** |
| 1024 | **2.53 M rec/s** | 252 K rec/s | **10.0×** |

The loop floor at ~250 K rec/s is the mpsc round-trip cap (~4 µs per record).
Batching saves (N − 1) round-trips and lowers to a single
`Log::append_batch` call; the curve flattens by N = 128 and is fully
amortized by N = 1024.

## Headline #2 — LZ4 is faster than None for redundant payloads

At batch = 1024, 256-byte all-zero values:

| Compression | Throughput | Per record | Notes |
|---|---:|---:|---|
| None | 2.53 M rec/s | 395 ns | baseline |
| **Lz4** | **3.34 M rec/s** | **300 ns** | wins because 256 B of zeros compresses to ~10 B → less disk I/O |
| Zstd | 1.13 M rec/s | 884 ns | compression cost dominates at this record size |

At larger sizes Zstd pulls ahead (`partition_append_single_zstd/1048576` is
**5.7 GiB/s**, slower than Lz4's 4.3 GiB/s at the same size, but with much
better compression ratio).

## Headline #3 — Default `LogConfig` is near-optimum

`preset_configs` group (256 B value, single `append()`):

| Preset | Throughput | vs default |
|---|---:|---:|
| `default` | 244 K rec/s | baseline |
| `throughput_oriented` | 254 K rec/s | **+4%** |
| `small_footprint` | 165 K rec/s | **−32%** |

`throughput_oriented` (256 MiB segments, 32 KiB index interval, 1 MiB
batch ceiling) buys only 4 % over defaults — the defaults are well-tuned
for single-producer workloads. `small_footprint` (1 MiB segments, 1 KiB
index interval) measurably hurts because rotation pressure dominates.

### `segment_size_threshold` sweep (256 B value, single `append()`, none)

| threshold | mean time | throughput |
|---:|---:|---:|
| 64 KiB | 11.2 µs | 89 K elem/s |
| 1 MiB | 19.2 µs | 52 K elem/s |
| 16 MiB | 17.8 µs | 56 K elem/s |
| 64 MiB | 16.4 µs | 61 K elem/s |
| 256 MiB | 17.6 µs | 57 K elem/s |

**Read carefully** — this cell shows the **counter-intuitive direction**
(64 KiB faster than larger). Theory predicts more rotations → more
fsyncs → slower, so 64 KiB should be the slowest. The CIs are wide
(11.0-11.4 µs for 64 KiB vs 17.6-20.8 µs for 1 MiB) and we did not isolate
the cause — this is one of the cells **dominated by variance on Windows
file I/O at the timescales involved (10-20 µs / iter)**. To get clean
signal here, the bench harness would need either a longer per-iter
workload (e.g., bench appending 1000 records and dividing) or a
Linux/Docker run.

## What this baseline does NOT tell us

- **HTTP-path throughput.** The figures here are library-only. The
  `kafko-http` HTTP harness has its own bench tooling at
  `scripts/kafko_docker_bench.ps1`.
- **Tail latency.** Criterion reports mean + CI, not p95 / p99. For
  latency distributions, use the `kafko-bench` matrix harness.
- **`Drop` shutdown durability cost.** Not benched here — covered by the
  `drop_without_shutdown_still_fsyncs_on_multi_thread_runtime`
  integration test only.

## Outstanding tunables not yet swept

- `batch_max_bytes` / `batch_max_records` — only matters under concurrent
  producer load (these gate natural batching from the writer task's
  inbox). The bench would need a concurrent-producer harness to surface a
  signal. Deferred until the v0.2 multi-partition work, which is where
  contention becomes interesting.
- Multi-partition fan-out — N/A in v0.1 (single partition per topic).
