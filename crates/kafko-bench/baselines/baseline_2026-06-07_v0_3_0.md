# kafko v0.3.0 baseline — 2026-06-07 (multi-partition)

Throughput snapshot after the multi-partition + key-routing change. Focus: did
the 0.3 rework regress the single-partition hot path, and does multi-partition
raise throughput? Measured with the `kafko-bench` harness (no hotpath
instrumentation), `KAFKO_PARTITIONS` knob added this session.

## Finding 1 — small single-send regression (~15 %), isolated to low batch sizes

`sequential` dropped from ~273 K (v0.2) to ~230 K median (v0.3) — roughly −15 %,
near the edge of the documented variance band but consistently on the low side
across two independent runs (245 K and 230 K medians).

**Cause: the per-wake-up `Notify` pulse.** The merged consumer wakes on a
`tokio::sync::Notify` shared across a topic's partitions; the partition writer
calls `wake.notify_waiters()` once per writer wake-up. `notify_waiters()`
acquires the Notify's internal waiters mutex **even when no consumer is parked**.

The regression scales inversely with records-per-wake-up, which fits exactly:

- `sequential`: 1 record per wake-up → 1 mutex acquire **per record** → visible ~15 % hit.
- `concurrent` / `batch`: many records coalesced per wake-up → the pulse is
  amortized across the whole natural/explicit batch → no measurable change
  (concurrent p1 ≈ 910 K, batch ≈ 2.3 M, both within v0.2 variance).

**Mitigation — APPLIED 2026-06-07.** A `Wakeup { notify, parked: AtomicUsize }`
now backs the shared signal (`crates/kafko/src/topic.rs`). A consumer increments
`parked` before its tail re-scan + park; the writer does
`if parked.load(Relaxed) > 0 { notify.notify_waiters() }`, so the hot path pays a
relaxed atomic load instead of a mutex acquire, and skips the pulse entirely when
no consumer is parked. It's lost-wakeup-safe because the consumer's re-scan routes
through the writer task's inbox, so the channel send→recv edge guarantees the
writer observes the `parked` increment before any later append it would pulse for
(full reasoning in `Wakeup`'s doc comment).

**Result:** `sequential` median 230 K → **247 K** and the floor 208 K → 233 K —
the whole distribution shifted up. The residual gap to the v0.2 ~273 K is within
the documented ±20–30 % single-record variance band; the recoverable
mutex-on-hot-path cost is gone. `batch`/`concurrent` also ticked up (within noise).

## Finding 2 — multi-partition does NOT raise single-node throughput

`concurrent` is flat-to-slightly-down as partitions increase (p1 ≈ p4 ≈ 910 K,
p8 ≈ 829 K). Multi-partition is **not** a throughput optimization on a single
disk under this closed-loop workload, for the same reason the BufWriter
experiment was rejected (see memory `project-bufwriter-refuted`):

- With **1 partition**, the single writer's *natural batching* coalesces all 16
  producers' queued sends into large `write()`s — already near-optimal syscall
  amortization.
- With **N partitions**, each partition's writer sees a thinner stream (≈16/N
  producers), so batches shrink and total `write()` count rises; and N writer
  tasks oversubscribe 4 cores + contend on one disk.

Multi-partition's value is **ordering-scope** (per-key order + parallel-safe
writes) and being the **prerequisite for consumer groups** — not raw single-node
write throughput. On a multi-disk / multi-consumer-group deployment the parallel
writers would matter; this single-disk closed-loop bench can't show that.

## How to regenerate

```powershell
cargo build --release -p kafko-bench
$bin = "target\release\kafko-bench.exe"
$env:KAFKO_TOTAL_RECORDS = "100000"; $env:KAFKO_RESET = "1"
# vary scenario + partition count:
$env:KAFKO_SCENARIO = "concurrent"; $env:KAFKO_PARTITIONS = "8"; & $bin
```

`KAFKO_PARTITIONS` (default 1) was added to `crates/kafko-bench/src/main.rs` this
session; it applies to every scenario's topic via
`create_topic_with_config_and_partitions`.

## Not covered here

- Per-function timing/alloc (run `scripts\kafko_hotpath_matrix.ps1` for that;
  it would localize the Notify-pulse cost to the writer loop).
- The criterion `send_batch` / `partition` / `config_sweep` matrices — the
  `partition` bench exercises `Partition` directly (no `wake`, so unaffected by
  the pulse); `send_batch` goes through the new Producer→Topic path and is the
  right place to A/B the mitigation above once applied.
