# kafko v0.2.0 baseline — 2026-05-29

Captures the v0.2.0 measurement set: criterion `send_batch` matrix +
hotpath allocation snapshot for the new `lz4_sequential` scenario. **Criterion
data also pinned under `target/criterion/<group>/<param>/v0_2_0/`** via
`cargo bench --features compression-all -- --save-baseline v0_2_0` so future
runs can A/B against it with `cargo bench -- --baseline v0_2_0`. The
v0.1.1 baseline is preserved in the same tree as `v0_1_1` for direct
side-by-side comparison.

## Headline #1 — LZ4 per-record allocation eliminated

The dominant v0.2.0 change. Measured via `.\scripts\kafko_hotpath_matrix.ps1`
which runs `kafko-bench` under `hotpath hotpath-alloc compression-lz4`. The
new `lz4_sequential` scenario mirrors `sequential` (1 task x 100 000
`send()` calls) under `Compression::Lz4`.

| Source | `compression::compress` total alloc | Per-call avg |
|---|---:|---:|
| v0.1.1 (`lz4_flex 0.11`, default `compress_into`) — historical baseline | ~1.2 GiB across 311 k LZ4 calls (~93 % of total heap traffic) | ~4 KiB |
| **v0.2.0 (`lz4_flex 0.13`, `compress_into_with_table`)** — this run | **24.9 KiB across 100 k LZ4 calls (0.10 % of total)** | **~0 B** |

Reduction: ~99.997 %. What remains is the floor — three thread-local
`HashTable4KU16` instances (one per encoder thread that touched LZ4),
allocated once on first use and reused for the rest of the process lifetime.

Sanity check — full process allocation under the same workload:

| Process metric (lz4_sequential, 100 000 sends) | v0.2.0 |
|---|---:|
| Total process heap traffic | 24.5 MB |
| `compression::compress` row | 24.9 KiB (0.10 %) |
| `partition::append` row (mpsc command alloc) | 8.4 MB (34 %) |
| `log::append_batch` row | 6.9 MB (28 %) |
| `producer::send` row | 3.1 MB (12 %) |

Memory traffic on the LZ4 path is now indistinguishable from the no-codec
path (`sequential` scenario shows the same 24.6 MB process total).

## Headline #2 — `send_batch` vs loop of `send()` (256 B values, no compression)

Same shape as v0.1.1's headline #1. Cells show median throughput; the
v0_1_1 column reads from the pinned baseline in `target/criterion/`.

| N | v0.1.1 (pinned) | **v0.2.0** | Loop of N x `send()` (v0.2.0) | Speedup vs loop |
|---:|---:|---:|---:|---:|
| 1 | 64 K | 267 K | 273 K | 1.0x |
| 8 | 376 K | 1.14 M | 276 K | **4.1x** |
| 32 | 717 K | 1.61 M | 281 K | **5.7x** |
| 128 | 1.14 M | 2.92 M | 286 K | **10.2x** |
| 1024 | 1.72 M | **3.66 M** | 274 K | **13.3x** |

The pinned v0_1_1 baseline measurements were captured during an earlier work
session and are lower than the v0.1.1 README's published figures (which used
better-isolated runs). Treating the **v0_2_0 column** as the
current-state-of-this-machine for v0.2.0 is the honest read.

The single-`send` loop floor of ~275 K rec/s is the mpsc actor round-trip
(~3.6 µs per record); batching saves (N - 1) of those round-trips and lowers
to one `Log::append_batch` call. Curve flattens by N = 128, fully amortized
by N = 1024.

## Headline #3 — `send_batch` with compression (256 B all-zero values)

| Compression | N = 1 | N = 128 | N = 1024 |
|---|---:|---:|---:|
| None | 267 K | 2.92 M | 3.66 M |
| **Lz4** | 286 K | 3.01 M | **4.05 M** |
| Zstd | 241 K | 1.06 M | 1.15 M |

LZ4 is now slightly faster than None at every batch size, not just large ones
— the per-call hash-table alloc that previously made small-batch LZ4 a wash
is gone in v0.2.0. The compressed-output-is-smaller win at large batches
remains (`Lz4 1024` at 4.05 M vs `None 1024` at 3.66 M = +11 %).

Zstd numbers are essentially unchanged from v0.1.1 — no zstd code changes
in v0.2.0.

## Headline #4 — Multi-size single-`send()` throughput (kafko-bench, no hotpath)

Run via `.\scripts\kafko_lib_multisize_bench.ps1`. Single producer tight loop,
one `producer.send().await` per record, varying value size and codec.
KAFKO_VALUE_SIZE / KAFKO_TOTAL_RECORDS / KAFKO_SCENARIO env vars drive the
binary; the wrapper iterates 6 sizes x 3 codecs and writes results to
`scripts\tmp\kafko_lib_multisize_<ts>\results.txt`.

### records/sec

| Size | none | lz4 | zstd |
|---|---:|---:|---:|
| 64 B    | 282,619 | 285,377 | 238,996 |
| 256 B   | 273,199 | 286,711 | 238,283 |
| 1 KiB   | 216,075 | 267,409 | 242,199 |
| 4 KiB   | 120,315 | 260,233 | 228,961 |
| 128 KiB |  11,128 |  51,606 |  52,127 |
| 1 MiB   |   3,032 |  10,561 |   5,283 |

### MiB/s value bytes

| Size | none | lz4 | zstd |
|---|---:|---:|---:|
| 64 B    |    17.2 |    17.4 |    14.6 |
| 256 B   |    66.7 |    70.0 |    58.2 |
| 1 KiB   |   211.0 |   261.1 |   236.5 |
| 4 KiB   |   470.0 | 1,016.5 |   894.4 |
| 128 KiB | 1,391.0 | 6,450.8 | 6,515.8 |
| 1 MiB   | 3,031.7 | **10,561.2** | 5,283.3 |

LZ4 beats `None` at every size on the single-send path in v0.2.0. At small
records the per-call hash-table alloc that hurt v0.1.1 is gone; at large
records the smaller on-disk write more than pays for the compression CPU.
The 4 KiB cell is where LZ4 first pulls clearly ahead (2.2x vs None).

## What this baseline does NOT tell us

- **Tail latency.** Criterion reports mean + CI, not p95 / p99. For per-cell
  HTTP latency see the Docker bench output at
  `scripts/tmp/kafko_docker_bench_results_<ts>.txt`.
- **Concurrent producer scaling.** The multi-size scenarios are single-producer.
  See the `concurrent` scenario in the hotpath matrix for the contention shape.
