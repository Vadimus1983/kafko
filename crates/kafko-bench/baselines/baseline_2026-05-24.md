### Derived figures

- Producer wrapper overhead = `send` total - `append` total = **~200 ms / 467k calls** -> wrapper is effectively free.
- Inner profiled work per send (amortized over the 6.74-record batch):
  - `record::encode_with`           ~907 ns
  - `compression::compress`         ~543 ns (311k/467k * 814 ns)
  - `segment::append`               ~785 ns (5.29 us / 6.74)
  - `flush_append_batch` overhead   ~2.44 us (18.46 us / 6.74 - segment::append)
  - **Accounted: ~4.5 us**
  - **Unaccounted gap: ~42 us per send** -> mpsc send + writer wakeup + oneshot reply + producer wakeup
- Natural batch size = 467,007 / 69,291 = **6.74 records / flush** (well below `batch_max_records = 1024` and `batch_max_bytes = 64 KiB` caps).
- `segment::sync` fired once at 128.94 ms -- the shutdown fsync. Insignificant in aggregate, but a real p99 spike when it lands.

### Notable

- `encode_with` allocates zero -- it writes into the caller's `BytesMut`.
- `compress` allocates 6.4 KB per call -- the freshly-allocated `Vec<u8>` returned by
  `zstd::bulk::Compressor::compress(raw)` (compression.rs:57) and by
  `lz4_flex::compress_prepend_size(raw)` (compression.rs:46). Reusing a thread-local
  output buffer is the headline win this snapshot exists to track.
- `append_batch` allocates 15.3 KB per flush -- the `BytesMut` (~7 KB for a 6.74-record
  batch) plus three small `Vec`s (`sizes`, `actual_sizes`, `offsets`). One of those
  (`sizes`) is dead code after compression-aware sizing landed (log.rs:160, ignored at
  line 204).
- `partition::append` 88 B/call = `PartitionCommand::Append { record, reply }` envelope
  boxed into the bounded mpsc slot.
- `producer::send` 32 B/call = the `oneshot::channel()` allocation per request.

| `main`, `hp-server`, `hp-mcp`, 5x unnamed | ~0% | <10% | Idle |

The 4 worker threads averaging <1% CPU each across 82 s of bench load confirms the
hot path is **wait-bound, not compute-bound** -- the ~42 us gap in `producer::send`
is scheduling latency on the actor round-trip, not CPU work.

## Buckets that returned empty

`futures`, `channels`, `streams`, `gauges`, `dbg_entries`, `val_entries` --
nothing in kafko is wrapped with `hotpath::measure_future!()`, `hotpath::channel!()`,
or the dbg / val macros. `tokio_runtime` requires `hotpath::tokio_runtime!()` in the
bench main and is not yet wired.

If we want runtime metrics later (park count, busy duration, queue depths, blocking
threads), that is a one-line add in `kafko-bench/src/main.rs`.

## How to regenerate this snapshot

```powershell
# Reset the data dir each run so baselines are not contaminated by prior segments.
$env:KAFKO_RESET = "1"
cargo run --release `
  -p kafko-bench `
  --features hotpath,hotpath-alloc,hotpath-mcp
```

Then query the MCP server via the `hotpath` tools (`functions_timing`,
`functions_alloc`, `threads`, `profiler_status`) and update the tables above.

## Optimization targets this baseline is meant to track

1. Reuse `compress` output buffer (compression.rs:43-61) -- expected: ~1.9 GB drop in
   total allocations, no change to wire format, no behavior change. Look for
   `compress` `alloc avg/call` going from 6.4 KB to a small constant.
2. Reuse `Log::append_batch` scratch buffers (log.rs:155-206) -- expected: ~1.0 GB
   drop in total allocations. Look for `append_batch` `alloc avg/call` going from
   15.3 KB to a small constant.
3. `std::fs` swap + `BufWriter` -- expected: `segment::append` per-call timing drop
   and `flush_append_batch` per-call timing drop; natural batch size may climb above
   6.74 if the writer drains faster than the producer can refill.
4. `Producer::send_batch` -- expected: per-record timing in `producer::send` drops
   sharply (the ~42 us mpsc gap amortizes over N records); batch ratio rises to the
   batch caller's chosen N.
